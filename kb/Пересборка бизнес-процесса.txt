Фиксация источника лидов №1 — «Мониторинг»
Общая логика

Источник лидов называется «Мониторинг БКИ».

Суть:
мы получаем компании, которые уже обратились в банк за финансовым продуктом, и понимаем, что у них есть текущая потребность.

Как формируется мониторинг

1. Раз в месяц

Ты формируешь список примерно 20 000 ИНН.

2. Этот список отправляется подрядчику

Подрядчик:

ставит эти ИНН на мониторинг в БКИ

3. Что делает мониторинг

По этим ИНН регулярно проверяется:

были ли запросы банков в БКИ

то есть обращался ли клиент в банк

Если банк делает запрос в БКИ → это фиксируется.

Что мы получаем

Каждый день приходит отчет со списком компаний, по которым:

был запрос в БКИ

значит компания обращалась в банк

В отчете есть:

ИНН компании

сумма запроса

Как интерпретируется потребность

Менеджеры анализируют сумму:

Если сумма круглая (например):

10 000 000
5 000 000
25 000 000

→ скорее всего кредит

Если сумма не круглая

3 482 150
7 913 420
12 554 000

→ скорее всего банковская гарантия

Фильтр по сумме

Вы работаете только с запросами:

от 3 000 000 ₽

Все что меньше — не обрабатывается.
Источник лидов №1 — Мониторинг БКИ
1. Формирование базы для мониторинга

В конце каждого месяца формируется список примерно 20 000 ИНН.

Этот список передается подрядчику.

Подрядчик ставит эти ИНН на мониторинг в БКИ.

Смысл мониторинга:
отслеживаются запросы банков в БКИ, которые означают, что компания обращалась в банк за финансовым продуктом.

2. Получение сигналов

Каждый день приходит Excel-файл от подрядчика.

Файл получает:

либо ты

либо руководитель группы

В файле содержится

В исходном файле нет ИНН, есть:

ID компании

дата запроса

сумма

иногда тип продукта
(кредит / банковская гарантия)

Связка ID → ИНН

У вас есть отдельная таблица соответствия:

ID → ИНН

Поэтому процесс выглядит так:

1️⃣ приходит файл с ID
2️⃣ по таблице соответствия определяется ИНН

Что такое сигнал

Сигнал = обращение компании в банк

Важно:

по одному ИНН может быть несколько сигналов в день

потому что:

компания обратилась в несколько банков

или есть несколько финансовых продуктов

4. Логика работы Bitrix24 после загрузки

После импорта запускаются роботы Bitrix.

Проверка ИНН

Робот проверяет:

есть ли такой ИНН в системе

лидов

компаний

сделок

Сценарий 1 — ИНН уже есть в системе

Если ИНН найден:

1️⃣ создается задача ответственному менеджеру

В задаче фиксируется:

сигнал мониторинга

2️⃣ дополнительно создается дело (комментарий)
— как дублирование сигнала.

То есть менеджер получает:

Сценарий 2 — ИНН в системе нет

Если ИНН новый:

2️⃣ в лид ставится задача менеджеру

3️⃣ запускается робот телефонов

(или "сворк", как ты сказал)

добавляет их в карточку компании/лида.

Источник лидов №1 — Мониторинг БКИ
1. Общая логика источника

Мониторинг — это источник лидов, который позволяет выявлять компании, у которых прямо сейчас появилась потребность в финансовых продуктах.

Потребность определяется через запросы банков в БКИ.

Когда компания обращается в банк за:

кредитом

банковской гарантией

или другим финансовым продуктом

банк делает запрос в БКИ. Этот запрос фиксируется и используется как сигнал потребности.

2. Постановка компаний на мониторинг

В конце каждого месяца формируется список примерно 20 000 ИНН компаний.

Далее:

Список ИНН передается подрядчику.

Подрядчик ставит данные ИНН на мониторинг БКИ.

Система начинает отслеживать запросы банков по этим компаниям.

3. Получение сигналов

Каждый день подрядчик отправляет Excel-файл с результатами мониторинга.

Файл получает:

либо руководитель отдела продаж

либо один из руководителей группы.

4. Структура исходного файла

В исходном файле нет ИНН компаний.

Вместо этого в нем указаны:

ID компании

дата запроса

сумма запроса

иногда тип продукта
(кредит или банковская гарантия).

5. Определение ИНН

В компании имеется таблица соответствия:

ID → ИНН

Поэтому перед загрузкой в CRM выполняется сопоставление:

ID из файла мониторинга

определяется соответствующий ИНН.

6. Понятие сигнала

Сигнал — это факт обращения компании в банк.

Особенности:

по одному ИНН может быть несколько сигналов в день

это возможно если:

компания обратилась в несколько банков

либо подала несколько заявок на разные продукты.

7. Подготовка файла для CRM

После сопоставления ID и ИНН формируется единый Excel-файл, который содержит:

ИНН компании

дату запроса

сумму

тип продукта (если указан).

Этот файл массово загружается в Bitrix24.

8. Логика обработки в Bitrix24

После загрузки файла запускаются автоматические роботы CRM.

Проверка наличия ИНН

Система проверяет:

есть ли данный ИНН:

в лидах

в компаниях

в сделках.

Сценарий 1 — ИНН уже есть в системе

Если компания уже есть в CRM:

создается одна задача ответственному менеджеру

в задаче фиксируются все сигналы мониторинга по этой компании

дополнительно создается дело (комментарий) в карточке.

Задача выступает уведомлением о новой потребности клиента.

Сценарий 2 — ИНН в системе отсутствует

Если компания новая:

создается новый лид

название лида формируется автоматически:

Название компании / ИНН / Мониторинг

в лид ставится задача менеджеру с сигналом мониторинга

9. Подгрузка телефонов

После создания лида автоматически запускается робот, который:

обращается к системе ISwork

собирает телефонные номера

добавляет их в карточку лида/компании.

На первичном этапе используются:

номера из ISwork

номера, которые уже есть в CRM.

10. Назначение ответственного

Если создается новый лид, ответственным автоматически назначается:

руководитель отдела продаж.

После этого лид вручную распределяется между менеджерами.

11. Распределение сигналов
Если сигнал по существующему клиенту

Задача автоматически назначается:

тому менеджеру, который закреплен за клиентом.

Если сигнал по новому клиенту

Лид распределяется:

вручную между менеджерами отдела продаж.

Если сигнал пришел на карточку уволенного менеджера

Лид также перераспределяется вручную между действующими менеджерами.

12. Объем сигналов

Средняя нагрузка:

400–450 сигналов в день

что соответствует примерно:

200–250 компаниям (лидам).

Причина:
по одной компании может быть несколько сигналов.

13. Правило обработки

Все сигналы мониторинга, полученные в текущий день, должны быть:

разобраны менеджерами в течение этого же дня.

Важно

Алгоритм дальнейшей работы менеджеров:

анализ компании

звонки

количество попыток

работа в мессенджерах

будет описываться отдельно в бизнес-процессе продаж и не входит в описание источника лидов.
Дополнение к источнику лидов «Мониторинг»
14. Подготовка файла для CRM

На текущий момент подготовка файла для загрузки в Bitrix24 выполняется вручную.

Процесс выглядит так:

Подрядчик отправляет Excel-файл с сигналами мониторинга.

Файл получает:

руководитель отдела продаж

либо руководитель группы.

В исходном файле содержатся ID компаний, а не ИНН.

Руководитель вручную:

сопоставляет ID компаний с ИНН
используя таблицу соответствия
ID → ИНН

После сопоставления формируется единый Excel-файл, содержащий:

ИНН компании

дату запроса

сумму запроса

тип продукта (если указан).

Подготовленный файл вручную загружается в Bitrix24 через массовый импорт.

Важная фиксация для будущего ТЗ

В текущей модели есть ручной этап подготовки файла, который включает:

сопоставление ID → ИНН

формирование финального файла для CRM

загрузку в Bitrix24.

Этот этап потенциально может быть автоматизирован в будущем бизнес-процессе.
